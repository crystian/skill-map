Three coordinated landings on top of `59eb351`:

1. **Dockerfile bug fixes**. Two latent regressions from the earlier reorganization (`1f24334`), neither caught by the existing test suite at the time. (a) `npm ci` was failing because the root `workspaces` array enumerates seven packages but the Dockerfile was only copying three of their `package.json` manifests; added explicit `COPY` lines for `testkit/`, `e2e/`, `web/`, `examples/hello-world/`. (b) `web/scripts/build-demo-dataset.js` reads `fixtures/demo-scope/` from the repo root, but after the fixtures relocation that directory was no longer in the Docker build context (it used to piggyback on `COPY ui/`); added an explicit `COPY fixtures/demo-scope/ ./fixtures/demo-scope/` before the dataset build. Two new structural assertions in `src/test/dockerfile-demo-assets.test.ts` cover both regressions.

2. **Railway deploy workflow with path filter**. New `.github/workflows/deploy-web.yml` replaces the GitHub-Railway auto-deploy integration with a path-gated workflow: only pushes touching `web/`, `ui/`, `spec/`, `fixtures/demo-scope/`, `Dockerfile`, `Caddyfile`, or root `package.json`/`package-lock.json` trigger a deploy. Two-job structure (`read-version` reads `web/package.json` runtime, `deploy` uses that as its job name so the GitHub Actions UI labels each deploy with the web version). Railway CLI exact-pinned (`@railway/cli@4.6.0`). Concurrency group prevents overlapping deploys. The setup steps the user does once (generate `RAILWAY_TOKEN`, disconnect the GitHub-Railway integration, confirm the service name) are documented in `context/scripts.md`.

3. **`@skill-map/web` versioning + landing footer**. The private web workspace gets its own version (`0.0.0` → `0.1.0`) and a seed `CHANGELOG.md`. The footer now carries three labelled tags with two distinct policies: `spec v…` and `web v…` are build-time substitutions in `web/scripts/build-site.js` (the site serves the schemas itself, so the footer must match what's served; the web version is trivially correct at build time); `cli v…` is a runtime fetch against `https://registry.npmjs.org/@skill-map/cli/latest` from `web/app.js` (the site doesn't ship the CLI binary, so the footer reflects "currently published on npm", best-effort with a `cli v—` placeholder if the registry is unreachable). `AGENTS.md` and `CONTRIBUTING.md` extend the changeset gate to include `web/`; `context/scripts.md` documents the deploy filter and the version policy.

Empty changeset: no published surface changes. `@skill-map/spec`, `@skill-map/cli`, and `@skill-map/testkit` are untouched. The `@skill-map/web` `0.0.0 → 0.1.0` bump is the one-time formal initialization of the workspace's versioning — set manually here rather than routed through the changeset machinery so the next Version Packages PR doesn't re-bump it to `0.2.0` unintentionally. Per the `.changeset/README.md` "non-semver-relevant" guidance, this commit ships an empty changeset to satisfy the `check-changeset.js` CI gate without bumping any version. Future `web/` changes ship their own changeset against `@skill-map/web` as normal.
